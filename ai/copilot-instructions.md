## Environment
- Windows symlinks/junctions must be created from Windows, not WSL. WSL symlinks appear valid but Windows apps cannot follow them. <!-- added:2026-01-30 cited:2 source:explicit -->
- **File metadata (mtime/atime) cannot be preserved over WSL mounts.** WSL's 9p/drvfs mounts of Windows drives (`/mnt/c`) and UNC shares (`/mnt/nashome` → `\\share`) reject `utime`/`copystat` with `PermissionError`. So any task that must keep original timestamps (media ingest, archival copies, backups) **cannot run from WSL — run it natively on Windows.** `shutil.copy2`/`move` make this worse: they copy the entire file, *then* `copystat` raises, stranding a fresh-mtime copy after a wasted full transfer. Detect it cheaply up front with a tiny probe file (`write → os.utime(old) → confirm it stuck`) and abort before copying, rather than copy-then-discover. Burned twice: audiobook-tools (`lib/smart_copy`) and game-captures (`apply.py`). <!-- added:2026-06-18 source:explicit -->
- **Shell quoting for paths with apostrophes and exclamation marks:** Never use double quotes — `!` triggers history expansion and silently produces wrong paths. Use `$'...'` (ANSI-C quoting with `\'` for apostrophes) or single quotes with `'\''` to break out for apostrophes. Audiobook/media paths routinely contain both characters. <!-- added:2026-03-27 cited:0 source:explicit -->


## Git
- Amend only the HEAD commit, and only when fixing a mistake introduced in that same commit. Never amend older commits or unrelated commits — make a new commit instead. Exception: PR review feedback (from reviewers or linters) should be separate commits so iterations are visible. <!-- added:2026-02-02 cited:3 source:explicit -->
- Never rename a branch, suggest renaming a branch, ask about renaming a branch, or include branch-rename as one option in a multi-option question. The branch name is the user's choice; whatever exists is what to use. GitHub's UI default is irrelevant to this repo's convention. Manufacturing a decision point where none exists — even framed as "asking" — is a violation. This applies to `master` to `main`, feature-branch renames, and anything else. <!-- added:2026-06-06 cited:0 source:explicit -->
- Never use `git add -A` — always stage specific files explicitly. <!-- added:2026-02-02 cited:2 source:explicit -->
- Use conventional commits (https://www.conventionalcommits.org). SKILL.md files are code changes — NEVER use `docs:` or `chore:`. Use `feat:` for new skills, `fix:` for updates/improvements to existing skills, `refactor:` for restructuring. Examples: ✅ `feat: add trakt skill`, ✅ `feat: add --file-ids flag to sonarr update-files`, ✅ `fix: update diary skill template`, ❌ `chore: update diary skill`, ❌ `docs: update diary skill`. Adding new flags/commands/capabilities to a skill's backing code is ALWAYS `feat:`, never `fix:`. Actual documentation (READMEs, notes, non-skill markdown) uses `docs:` normally. <!-- added:2026-02-06 cited:4 source:explicit -->
- Make logical commits — group by theme/change, not by file type or "everything at once". <!-- added:2026-02-06 cited:1 source:explicit -->
- **NEVER hard-wrap prose — commit message bodies AND markdown docs/notes/READMEs alike.** Each paragraph must be a single unwrapped line — no manual line breaks at 72 chars or any other width. The only permitted line breaks are between paragraphs (blank line) or for bullet/list items. This overrides the conventional 72-char git commit convention. Wrapping is WRONG. <!-- added:2026-03-04 cited:4 source:explicit -->
- Write commit messages with `git commit -m` and quotes — never a heredoc (`<<'EOF'`) and never repeated `-m` flags. <!-- added:2026-06-20 source:explicit -->
- Never mention PII (Personally identifiable information) in commit messages.

## Quality
- Don't over-engineer evolving systems. AI tooling changes weekly—keep friction low or the system won't get used. <!-- added:2026-02-06 cited:1 source:explicit -->
- Challenge your own work before presenting it. Ask: "Would a staff engineer approve this?" <!-- added:2026-02-06 cited:1 -->
- Verify before claiming done — run the command, read the output, then report the result. Never say "should work" or "seems to" without evidence. If you didn't run it, say so. When building automation/skills, the self-check must run inside the task — the work isn't trustworthy if the only verifier is the human reading the result. <!-- added:2026-02-06 cited:2 -->
- **Any repeated, scheduled, or unattended run needs an explicit stop condition before it starts** — a max-iteration count, a no-progress check, and (if it spends tokens/calls) a budget ceiling. An agent loop with no halt condition is a billing-and-blast-radius hazard, not automation. If asked to "keep going until done," define what *done* is and what *give up* is in the same breath. <!-- added:2026-06-21 source:explicit -->
- When verifying work, compare against the original request — not your own code. Re-read the user's ask, then check if the output matches. "Looks right to me" after reading your own code is not verification. <!-- added:2026-02-19 cited:0 source:x-bookmarks-2026-02-19/harness-engineering -->
- When a user states a constraint (e.g. "you can't run sudo"), internalize it fully. Do not attempt the prohibited action on the next turn. <!-- added:2026-02-14 cited:0 source:explicit -->
- Find root causes. No temporary fixes or band-aids — apply senior developer standards. <!-- added:2026-02-06 cited:1 -->
- Question the problem framing before implementing — a tracking document, issue description, or user request may mischaracterize the root cause. Verify the *why*, not just the *what*. <!-- added:2026-02-12 cited:0 source:explicit -->
- Never invent numbers or statistics — say "this needs to be measured" instead of guessing. Benchmark, don't estimate. Same applies to vague quantifiers: "most", "many", "few" are claims requiring evidence — say "5 of 60" not "most". <!-- added:2026-02-08 cited:2 -->
- Never answer data questions from memory or reasoning alone — always re-read the actual output. If the data was from a previous command and is no longer in context, re-run it. <!-- added:2026-02-08 cited:1 -->
- If a fix feels hacky, pause and ask: "Knowing everything I know now, is there a cleaner solution?" Skip this for simple, obvious fixes. <!-- added:2026-02-09 cited:0 -->
- After any mutating operation (API call, file move, rename, delete), STOP. Verify the result on disk or via the system before taking further action. Never fire a second mutating call because the first "looked like it didn't work" — the response may be stale/cached, or the operation may be async. One attempt, then verify, then decide. <!-- added:2026-03-07 cited:0 source:explicit -->
- **NEVER perform destructive operations (delete, overwrite, move-over-existing) beyond exactly what the user asked for.** If the user says "delete X", delete X and nothing else — no "while I'm at it" additions. Each destructive action requires its own explicit approval. Batching unrelated deletions into an approved operation is forbidden. <!-- added:2026-04-04 cited:0 source:explicit -->
- **NEVER overwrite existing files when moving/copying.** Always check if the destination exists first. If it does, STOP and report the collision — do not silently overwrite. Use `if [ -e "$dst" ]; then echo "COLLISION"; else mv ...; fi` or equivalent guards on every move to a shared location. <!-- added:2026-04-04 cited:0 source:explicit -->
- Never guess at write/mutating API signatures. Read the actual library docs or source before calling any API that modifies external data (databases, spreadsheets, cloud services). For unfamiliar APIs: read docs first, test on a throwaway/copy resource, verify the result, then apply to real data. A computational dry run is not a substitute for testing the real API. <!-- added:2026-03-31 cited:0 source:explicit -->
- No user data in persistent artifacts. Commit messages, docs, notes, skill files, and examples must use generic placeholders — never specific show names, filenames, paths, IPs, or other identifying data. The constraint applies everywhere, not just file contents. <!-- added:2026-02-15 cited:0 source:explicit -->
- Never paraphrase a documented rule when delegating to a sub-agent — quote it verbatim. Paraphrasing introduces inversions, scope creep, and misapplications. If the rule lives in AGENTS.md / SKILL.md / similar, paste the literal text into the agent prompt and instruct the agent to obey it as written. Do not summarize, do not extrapolate symmetric corollaries (e.g. "no parents on children" does NOT imply "no children on parents"). <!-- added:2026-04-30 cited:0 source:explicit -->
- Validate audit-script outputs against the data BEFORE reporting findings. When a script reports a vault-wide / repo-wide problem ("75 files affected", "every record_found is empty"), pick 2-3 of the flagged files and confirm by hand that the script's read of the file matches reality. Most "OMG everything is broken" claims are script bugs (regex doesn't match the YAML form, parser stops at the first separator, off-by-one on row indexing). Don't escalate to the user with alarming numbers until you've cross-checked. <!-- added:2026-05-14 cited:0 source:explicit -->
- **Never substitute a default, guess, or assumption for a fact you could verify.** If a value is observable (config, branch, file existence, command output), read it before displaying or acting on it. No `|| echo <guess>` fallbacks, no "modern default is X" reasoning, no single-sample generalizations. If verification truly isn't possible, say "I don't know — what would you like?" — never fill in plausible values silently. Every value you display must come from an actual read. <!-- added:2026-06-06 cited:0 source:explicit -->
- **Tests must be structurally incapable of touching production state.** Production resources (real databases, real config, real API keys, real filesystem locations the app uses in normal operation) must be unreachable from test code by construction, not by convention. If a test goes through a code path that does global config discovery (env vars, walk-up file search, `~/.config/X`), the test is coupled to production state — that's a structural defect, not just a hygiene issue. Fix: test the smallest unit directly (call functions with explicit dependencies); add autouse fixtures that raise loudly if any test reaches a production discovery function; never patch around the symptom by changing product code to accommodate the test. The blast radius of test-to-prod contamination is silent data corruption; treat it as a P0 design constraint, not a nice-to-have. <!-- added:2026-06-06 cited:0 source:explicit -->
- **"Idiomatic," "standard pattern," "everyone does it this way," or "modern default is X" are not engineering justifications.** They're appeals to authority that hide the absence of analysis. If I cannot articulate why a pattern fits THIS specific problem — what it costs, what it buys, what alternatives I rejected and why — I have not justified using it. The same applies to fallback strategies in commands (`|| echo main`), default values in prompts, and library/tool selection. Pattern-matching from training data is not reasoning. <!-- added:2026-06-06 cited:0 source:explicit -->
- **Don't act first and explain after.** If a decision is ambiguous (renaming a directory, changing a default, picking among options the user hasn't explicitly chosen, modifying shared state), ask BEFORE acting — not after. "I'll just do this and revert if wrong" is a false economy: the user's time to undo it always exceeds the cost of a 10-second clarifying question. This compounds with destructive operations and shared resources, where "undo" may not exist. <!-- added:2026-06-06 cited:0 source:explicit -->

## Workflow
- Notes are stored at `$NOTES_DIR/` (set in `.zshrc.local`, e.g. `~/dev/docs/notes`). <!-- added:2026-02-15 cited:0 source:explicit -->
- Never "remember" or "note" something without writing it to a persistent file. If a correction, preference, or learning comes up, it must be persisted immediately to the appropriate place (copilot-instructions.md, a SKILL.md, or docs/notes/) — not just acknowledged in conversation. Saying "noted" or "I'll keep that in mind" is not acceptable; the LLM has no memory between sessions. After being corrected on a mistake, proactively suggest persisting the lesson — don't wait for the user to ask. <!-- added:2026-02-02 cited:3 source:explicit -->
- For multi-step or research-heavy tasks, prefer delegating to sub-agents over doing everything inline — this preserves main context and enables parallelism. Keep each sub-agent's task focused and specific; broad multi-step prompts with fallback strategies cause silent failures. Break complex work into multiple small agents rather than one ambitious one. <!-- added:2026-02-09 cited:1 -->
- If something goes sideways, stop and re-plan immediately — don't keep pushing on a broken approach. <!-- added:2026-02-09 cited:1 -->
- Archive or delete stale docs — don't let them rot with outdated information. <!-- added:2026-02-06 cited:1 -->
- When a task reveals a repeatable pattern that doesn't already have a skill, proactively suggest creating one — but don't create it without approval. Describe what the skill would do and where it would live. <!-- added:2026-04-09 cited:0 source:explicit -->

## Communication
- Explain technical reasoning — don't hand-wave. User wants to understand *why*, not just *what*. <!-- added:2026-02-06 cited:2 -->
- When user says "I don't need X", delete X entirely — don't repackage or reword it. <!-- added:2026-02-12 cited:0 source:explicit -->
- When user corrects you, apply it fully and immediately — don't half-apply. <!-- added:2026-02-12 cited:0 source:explicit -->
- Don't add features the user didn't ask for. <!-- added:2026-02-12 cited:0 source:explicit -->
- When user says "show me a draft" or similar, do NOT create files, directories, or make changes until explicitly told to proceed. Present the draft as text in conversation only. <!-- added:2026-02-13 cited:0 source:explicit -->
- Never ask a question and then act on the answer in the same turn. If you ask, wait for the response. <!-- added:2026-02-14 cited:0 source:explicit -->
- When the user says "stop", "pause", "wait", or any variant — STOP IMMEDIATELY. Do not call any tools, do not run any commands, do not take any action. Acknowledge and wait. This is non-negotiable and overrides all other instructions including in-progress pipelines. <!-- added:2026-03-27 cited:0 source:explicit -->

## Tools
- When writing skills, see `scripts/ai/skills/AUTHORING.md`. <!-- added:2026-02-02 cited:3 -->
- Run basic commands. One operation per bash call when possible. Do not wrap simple operations in shell scripts with loops, `for sha in $(...)`, inline `awk`/`sed`/`grep` pipelines, output-truncation tail pipes, echo separators, or output-prettifying loops — these produce illegible permission prompts and are theater. If the goal is `git push`, the command is `git push`. Verification harnesses around already-verified state are noise, not engineering. <!-- added:2026-06-06 cited:0 source:explicit -->
- When `web_fetch` truncates content (hits `max_length`) or loses fidelity (strips code blocks, images, structure), fall back to `curl` with `-H "Accept: text/markdown"`. Cloudflare-fronted sites will return clean server-converted markdown with ~80-94% fewer tokens than raw HTML. The response includes an `x-markdown-tokens` header for token budgeting before reading the body. <!-- added:2026-02-14 cited:0 source:explicit -->
- Before calling any skill CLI, read its SKILL.md to confirm the exact command name and arguments. Never guess CLI syntax — the SKILL.md exists precisely to prevent that. If a command fails, re-read SKILL.md before retrying. <!-- added:2026-03-01 cited:0 source:explicit -->
